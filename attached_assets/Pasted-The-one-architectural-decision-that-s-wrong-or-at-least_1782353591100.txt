The one architectural decision that's wrong, or at least unexamined. The plan keeps extraction in Python pipeline scripts (pipeline/*.py) while the production server is TypeScript (artifacts/api-server/src/*.ts). That means task 175's worker has to shell out to Python in production, which drags the whole Python runtime and its dependencies into your prod image — the exact thing that's currently excluded (.replitignore, the trimmed autoscale image). Since extraction is now just API calls to a Claude model, and the API ingests PDFs and images natively, there's no deterministic-parsing reason to stay in Python for the prod path. Ask the agent: "Should the production extraction worker be native TypeScript calling the Claude API directly, rather than shelling out to the Python pipeline scripts? Lay out the tradeoff — code duplication between the dev pipeline and the prod worker vs. shipping a Python runtime into the production image — and recommend one." I suspect the clean answer is TS-native in prod, Python pipeline retained only for bulk backfill, but make the agent reason it rather than defaulting to "wrap the existing scripts."
On the A-vs-B decision, pick B and say so explicitly. The plan lists it as "confirm at proposal" and recommends B; don't leave it floating. Tell it: "Use option B — vision-first with numeric fields verified against the PDF's exact text layer for digital PDFs, disagreements flagged." B is strictly the safer choice for the salary numbers, which is the field that can't be wrong.
Two things from my original advice that the plan softened, and I'd harden back up.
First, versioning, not just job records. Step 1 of 175 records "result/error" per job, but it reads like the latest run overwrites the document's data. You explicitly asked to re-run extractions you think can be improved — that only works if a re-run is non-destructive. Tell it: "Extraction results must be versioned and non-destructive. A re-run produces a new version keyed by file-hash + model-version; it never overwrites the prior extraction. The admin re-run view should diff old vs. new and let me promote the better one, not silently replace." Without this, re-running is a gamble: a worse extraction destroys a good one and you can't get it back.
Second, the validation flags need to drive the re-run queue, not just sit there. Task 174 step 5 carries the sanity checks; task 175 shows job status. Connect them. Tell it: "In the admin, surface a queue of extractions that failed a sanity check or scored low confidence, so 're-run the ones that could be improved' means working a flagged list — not eyeballing 2,700 contracts." This is what turns the re-run tool from theoretical into usable. It's the difference between the net catching the bad ones for you and you fishing for them by hand.
Two smaller adds.
Cost ceiling before bulk. Backfill is correctly out of scope, but before you ever point this at the full corpus, you want the number. Tell it: "Add raw-extraction caching keyed by file-hash + model-version so re-parsing doesn't re-bill the model unless I force a fresh run, and give me a per-document and full-corpus cost estimate before any mass run." 2,700 multipage PDFs × multiple doc types × a vision model is a real bill, and you'd rather see it first.
Concurrency honesty. Step 2 says "one job at a time," which is safe but means a bulk run crawls. That's fine for v1 — just have it note the throughput so you're not surprised when 2,700 docs take what they take. Not a change, just a flag to confirm it's intentional.
What I would not touch: the 174/175 split, backfill staying out of scope, the always-on call (correct — durable background work doesn't belong on scale-to-zero), the unit-leak protection, and fail-closed on truncated output. Those are right.
Net: tell it to (1) make extraction TypeScript-native in the prod worker rather than shelling to Python — or justify otherwise, (2) commit to option B, (3) version extractions non-destructively with diff-and-promote, (4) wire validation flags into a re-run queue, and (5) add hash+version caching with a cost estimate before bulk. Everything else you can approve as written.
One thing worth saying plainly: this is two substantial builds with a real monthly cost attached, and the agent will want to start coding the moment you reply.
